详解React生命周期

发表于 2017-09-21 | 分类于 React |

生命周期概述

Life Cycle这一概念被广泛应用于诸多框架之中，它通常描述了某个事物阶段性发生变化的规律。在React中使用Life Cycle这一概念描述了组件存在于页面上的各种时期。

根据React官方的定义，我们可以大致将生命周期分为两大类：

  1. 组件被装载或卸载时
  2. 组件更新时

装载与卸载过程

组件装载与卸载过程在整个组件的生命周期内只会执行一次。

Mount-装载

装载过程中的生命周期如下

这里调用了componentWillMount与componentDidMount两个生命周期方法。分别是组件在执行即将装载时（render执行前）与装载结束(render执行后)的生命周期。

当在componentWillMount中执行setState时，由于在render执行之前发生所以组件只会更新自身的state，并不会触发第二次的render。而在componentDidMount中使用setState时则会触发组件的更新使得组件再一次被render造成重复渲染。

Unmount-卸载

组件的卸载只有一个生命周期即componentWillUnmount。通常被应用于清除定时器clearInterval，移除事件监听removeEventListener,防止由于闭包导致的内存泄漏。

组件更新过程

React的核心思想之一即View=F(Model),即通过数据Model层驱动视图View层的变化。而在React中通常通过两种形式改变Model的数据源：上层组件传递Props变化/自身执行了setState。这两种操作都会使组件触发更新相关的生命周期，并通过我们所制定的F()更新View层。

以上便是组件更新时可能遇见的生命周期，我们以执行一次setState为例，将依次执行shouldComponentUpdate => componentWillUpdate => componentDidUpdate三个生命周期。

shouldComponentUpdate

在React中，当一个父级节点改变其Props时，其下属的所有子级节点都会被重新渲染，而在一些情况下，对于子节点而言Props的改变并不需要重新进行render。所以我们可以在这个生命周期中自行判断当前传递的Props是否需要对当前组件进行更新，当在这个生命周期中返回false时，之后关于组件更新的生命周期不会被执行。

componentWillUpdate与componentDidUpdate

这两个生命周期与componentWillMount和componentDidMount十分相似，componentWillMount与componentDidMount代表着组件装载时render执行前后的时刻，而componentWillUpdate与componentDidUpdate分别代表组件更新时render执行的前后时刻。

在React官方文档中提到了不要在ComponentWillMount的过程中使用setState，原因也就十分清晰：当执行setState会触发shouldComponentUpdate => componentWillUpdate => componentDidUpdate下列生命周期，而在componentWillUpdate时使用即触发了无限递归调用导致爆栈。同理在shouldComponentUpdate中使用也会造成同样的问题。

componentWillReceiveProps

如果组件的更新时由Props变化导致的，则会在shouldComponentUpdate之前执行这个生命周期，在这个生命周期中使用setState不会触发二次渲染。通过这种方式我们可以实现通过外层的Props控制组件内部的State更新。

组件执行过程

在我们创建一个React组件时，除了会使用LifeCycle相关的函数外还会使用例如constructor(),render()类的函数，也会在一些位置初始化Props,State。那么一个真实的React组件全部执行过程是怎样的呢？

构造器阶段constructor

getDefaultProps: 在使用static defaultProps = {}时便是调用了内部getDeafultProps方法。如果在实例化时指定了props，则使用传入的props否则使用在static defualtProps中所定义的props。

组件Mounting阶段

getInitialState: 在使用this.state = {}时就是调用的内部的getInitialState方法。

componentWillMount: 判断组件中是否声明componentWillMount,有则执行，没有则直接执行render。而在componentWillMount中执行setState,会直接使当前的state与getInitialState合并避免二次渲染。

render: 递归渲染render的组件树。

componentDidMount: 判断有无componentDidMount,存在即执行。

由于render的递归特性，父组件的componetWillMount总是在子组件的conponentWillMount之前调用，而父组件的componentDidMount总是在子组件的componentDidMount之后调用。

组件Update阶段

当组件的props或state发生变化是，则会进入Update阶段。

componentWillReceiveProps: 判断有无componentWillReceiveProps，存在则执行。在此生命周期中执行setState是执行state合并不会触发render。

shouldComponetUpdate: 根据该函数返回值判断是否进行下面的操作。不指定则返回true。当该函数返回了false时，传入的nextProps与nextState仍然会替换之前的prevProps与prevState。

componentWillUpdate: 判断有无componentWillUpdate，存在即执行。

render: 递归渲染render组件树。

componentDidUpdate：判断有无componentDidUpdate，存在即执行。

同Mount阶段原理，父组件的componetWillUpdate总是在子组件的conponentWillUpdate之前调用，而父组件的componentDidUpdate总是在子组件的componentDidUpdate之后调用。

组件Unmounting阶段

componentWillUnmount: 在这个阶段组件会将state的更新队列重置为null。所以设置了setState也没什么卵用。

componentWillMount与componentWillUpdate

之所以把这哥俩放到一起说是因为这两个生命周期的地位十分尴尬，我们来逐一解析：

  1. 在componentWillMount时执行Ajax

    看完了上部分的解析，我们可以发现在componentWillMount中进行setState并不会触发rerender，而是会进行state合并才会继续执行render。那我们在这个生命周期里进行Ajax相关的请求并把数据存放到state中还不会触犯rerender岂不是美滋滋？然而现实是残酷的，由于Ajax的异步特性，我们并不能保证获取到数据之后进行setState的步骤一定会在render之前执行，也就导致了整个生命周期的不可控。

  2. 在componentWillMount是执行state初始化

  3. 在componentWillUpdate中执行setState

componentDidUpdate

这货可能是使用频率最高的生命周期…我们通常会在这个生命周期中进行Ajax操作。由于此时render内需要被渲染的组件已经全部Mount，我们也可以获取ref获取实例做一些羞羞的事情。

componentWillReceiveProps

我们可以在这个生命周期中得到prevProps与nextProps，于是我们可以Diff两者的区别处理一些需要做的事情：手动调用DOM方法，绘制Canvas等等。

还有一个更加常见的用例就是根据Props的变化维护自身的state。这在设计一个即是智能又是受控的组件时非常实用。

shouldComponentUpdate

上述文章中已经讲到,任何React父级组件的Update都会导致rerender。这里需要开发者自行判断传入的值是否需要进行更新。是不是十分蛋疼？没关系，React中除了基础的Component外，还为我们提供了PureCompoent。它帮我们自动完成了shouleComponentUpdate的生命周期：Diff前后的props与state变化，如果没有变动则不进行更行。

注意PureComponent在Diff时只进行浅比较，对于引用类型的比较显得无能为力

componentDidUpdate

当组件更新完成时，关于展示相关的数据可能已经更改，在这个生命周期中重新计算布局可能是一个不错的选择。

所谓冒泡排序就是顾名思义，做arr.length次数的循环，每次循环中都比较两两相邻的元素，让较大的元素浮上来，循环结束即排序完成。

其实复杂度为n^2的lowB算法还有选择排序、插入排序，但是实现的方式难度大同小异，这里选择一个比较有代表性的实现。

快速排序

非常经典的排序算法，引用了递归分治的思想。

  1. 从数组中随机选出一个值a作为参照，将比大于a的值放到右边的数组，将小于a的值放在左面的数组。

  2. 分别对左面的数组和右面的数组再次进行步骤一的操作。直到当数组仅剩一个值或没有值时结束操作。
